3 Pendekatan Arsitektur

Pilih satu — ini menentukan struktur folder, deploy strategy, dan kompleksitas

my-portfolio/
├── app/ # Next.js frontend
├── admin/ # Payload CMS
├── docker-compose.yml
└── shared/ # types, utils

A — Monorepo (Recommended)

Frontend + Payload CMS dalam satu repo. Shared types. Deploy via Docker Compose. Paling clean untuk solo dev.

Pros

  • Satu repo, satu deploy
  • Shared TypeScript types
  • Docker siap pakai

Cons

  • Sedikit lebih kompleks setup awal
portfolio-frontend/ # repo 1
└── Next.js + Tailwind
portfolio-cms/ # repo 2
└── Payload CMS

B — Separate Repos

Frontend dan CMS repo terpisah. Independent deploy. Lebih fleksibel tapi overhead koordinasi lebih tinggi.

Pros

  • Deploy independent
  • Swap CMS mudah

Cons

  • 2 repo dikelola
  • Type sync manual
my-portfolio/
├── app/
├── (frontend)/ # public pages
└── (payload)/ # CMS admin
└── payload.config.ts

C — Fully Integrated (Payload dalam Next.js)

Payload CMS dijalankan dalam satu Next.js app. Satu server, satu deploy. Paling simple tapi tightly coupled.

Pros

  • Satu server saja
  • Deploy paling simple

Cons

  • Frontend + CMS tightly coupled
  • Scale lebih sulit
